## Purpose

This document records the content strategy decisions for the post-call debrief screen. It defines **what** content the debrief contains, **who** produces each field (LLM vs backend), the **tone register**, and the **final JSON schema**. It is the single source of truth for debrief content — all other documents (architecture, epics, design specs) reference this.

## Context

The debrief screen design (Story 2.4) produced visual containers without validated content. The UX spec itself stated: *"Content direction to be defined first. Full content and section structure: TBD pending content strategy review."* This document resolves that TBD.

**Technical context:** The debrief is generated by a separate post-call API call to Qwen 3.5 Flash via OpenRouter with `json_schema` strict mode. The LLM receives the full conversation transcript and produces structured JSON. The backend merges LLM output with its own calculated fields to assemble the final debrief.

## Content Decisions (1-10)

### Q1: `strengths` / Points forts — REMOVED

**Decision:** No strengths field in the schema.

**Rationale:** The UX spec is unambiguous: *"Honesty builds trust, praise destroys it"*, *"No 'great job' for a 45% survival rate"*, *"The app never tells the user how to feel."* A `strengths: ["Good vocabulary range"]` field is praise dressed as objectivity. The survival percentage IS the positive feedback. The encouraging framing section (FR15b, >40%) provides data-driven context ("5% away", "+6% since last"), not opinions.

**PRD alignment:** No FR requires strengths. FR9-FR13 define the debrief content — strengths is not among them.

---

### Q2: `summary` — REMOVED

**Decision:** No summary field in the schema.

**Rationale:** A narrative summary ("You started strong but froze when the mugger escalated") is either redundant with individual sections or too vague to be actionable. The hero section (%, character, scenario, attempt #) IS the at-a-glance summary. Adding a text paragraph creates a "report card" register that contradicts the adversarial entertainment positioning. Each section (errors, hesitations, idioms, areas) is self-contained and specific — a general summary would dilute that specificity.

---

### Q3: Error `explanation` — REPLACED BY `context`

**Decision:** Each error has `user_said`, `correction`, `context`, `count`. No per-error grammar rule. The `context` field is a brief narrative note about WHEN in the conversation the error occurred.

**Rationale:**
- **Context aids memory anchoring** — "After the mugger's initial threat" connects the error to an emotional moment the user physically experienced. This makes the correction memorable.
- **Grammar rules are synthesized in `areas_to_work_on`** — no need to duplicate per error.
- **"Autonomous motivation" principle** — the correction shows WHAT to say, the context shows WHEN the user breaks down, and the areas give the study direction. Each layer serves a distinct role.

**Register:** Narrative brief, one sentence, third person. "After the mugger's initial threat" — not "Even I could see you choked there."

---

### Q4: `tips` vs `areas_to_work_on` — SINGLE FIELD: `areas_to_work_on`

**Decision:** Remove `tips`. Keep `areas_to_work_on` (2-3 items).

**Format:** Theme + actionable example in parentheses.
```
"Negative sentence structure (use don't/doesn't instead of 'not want')"
```

**Rationale:** The UX spec says *"Debrief ends with clear 'areas to work on'"* — not "tips." The product is a mirror, not a teacher. The debrief says "here's what's broken" (diagnostic), not "here's what to do" (instruction). The parenthetical makes the theme searchable: "English negative sentence structure don't doesn't" is a valid search query for self-study.

**Count:** 2-3 items strictly. Never 1 (too brief), never 4+ (no prioritization). The LLM MUST prioritize.

---

### Q5: Error Selection — MAX 5, DEDUPLICATED, WITH FREQUENCY

**Decision:** The LLM selects the 5 most significant errors maximum. Repeated errors are grouped (one card with count, not separate cards).

**Selection criteria (for LLM prompt):**
1. **Frequency** — error repeated 3x > unique error (it's a pattern, not an accident)
2. **Communication impact** — error that prevents comprehension > minor grammar mistake
3. **Diversity** — cover different problem types (structure, vocabulary, tense), not 5 variants of the same issue

**Rationale:** 15 error cards on mobile = information overload. "Constructive lucidity" requires selection. 5 well-chosen errors with frequency counts are more actionable than 15 cards in a wall. The `count` field compensates for deduplication.

**Edge case:** If the user makes 0-1 errors, show 0-1. The minimum is not 5.

---

### Q6: Hesitation — ARRAY OF 1-3, MINIMUM THRESHOLD 3 SECONDS

**Decision:** Schema changes from `longest_hesitation` (single object) to `hesitations[]` (array, max 3).

**PRD alignment:** FR12 says "longest hesitation **moments** and their **context**" — plural. The previous design showed only one moment. This corrects the discrepancy.

**Threshold:** Only silences > 3 seconds are reported. A 1.5s pause is a natural conversational gap, not a revealing hesitation.

**Backend/LLM split:**
- The backend measures durations from STT timestamps (end of character speech → start of user speech).
- The backend identifies the top 3 gaps > 3s and passes them to the LLM in the debrief prompt.
- The LLM adds narrative `context` for each gap.

---

### Q7: Error Context — INCLUDED

**Decision:** Keep the `context` field per error. One sentence, italic, muted in the UI.

**Rationale:**
- **Memory anchoring** — connecting errors to emotional moments makes corrections memorable.
- **Behavioral insight** — the user may notice patterns ("I make my worst errors when the character escalates").
- **Minimal design impact** — the context is already in the visual spec (italic, `#9A9AA5`, bottom of card).

**Constraint:** Maximum 1 sentence from the LLM. Brief, factual, situational.

---

### Q8: Error Frequency Count — INCLUDED

**Decision:** Add `count: integer` per error.

**Rationale:** The UX spec Moment 3 explicitly says *"You said 'I am agree' three times."* Repetition count IS debrief content. It reinforces "no hedging" — "you made this error 3 times, not once." It supports the deduplication strategy (Q5): one card with "x3" instead of three identical cards.

**Display rule:** Show count indicator only when `count >= 2`. A single occurrence doesn't need "x1."

---

### Q9: Tone — CLINICAL. NEUTRAL, FACTUAL, SPECIFIC. NO PERSONA.

**Decision:** The debrief is written in the app's voice, not the character's voice.

**Register:**

| Element | Register | Example | Anti-example |
|---------|----------|---------|-------------|
| Error correction | Factual, direct | "I don't want any trouble" | "A better way to say this would be..." |
| Error context | Narrative brief, 3rd person | "After the mugger's initial threat" | "This was when things got heated!" |
| Hesitation context | Factual, situational | "When asked to empty pockets" | "You totally froze here" |
| Idiom meaning | Clear definition | "I don't believe you" | "This fun expression means..." |
| Areas to work on | Diagnostic + example | "Negative sentence structure (don't/doesn't)" | "Try practicing negative sentences!" |
| Inappropriate behavior | Factual, no judgment | "The call ended because the conversation included inappropriate language" | "You shouldn't have said that" |

**Rationale:** The UX spec separates two registers: the character is sarcastic DURING the call, the debrief is an honest mirror AFTER. The Call Ended screen (Story 2.3) bridges with a theatrical phrase ("The mugger gave up on you"). The debrief shifts to analytical register. Adding `persona_comment` per section would mix entertainment and education in a space dedicated to education.

---

### Q10: Survival Score — BACKEND-CALCULATED, NOT IN LLM SCHEMA

**Decision:** `survival_pct` is computed by the backend, not generated by the LLM.

**Rationale:**
- The score is based on scenario progression (successful exchanges / expected exchanges, or call duration / target duration).
- The backend has this data (LiveKit session timestamps, scenario config).
- The debrief LLM receives the transcript AFTER the call — it doesn't have the conversation LLM's internal "patience meter" state.
- A backend calculation is **deterministic and reproducible**. An LLM score would vary across calls for the same performance.

**Suggested formula (for architecture):**
```
survival_pct = min(100, floor(successful_exchanges / scenario_expected_exchanges * 100))
```

The exact formula is an architecture decision (Epic 7, Story 7.1). For design purposes: it's an integer 0-100, calculated by the backend. **Uses `floor()` not `round()`** to prevent false 100% — the green "100%" display only appears when the user truly completes the entire scenario.

## Final JSON Schema — LLM Output

This is exactly what the LLM produces via `json_schema` strict mode:

```json
{
  "errors": [
    {
      "user_said": "I am not want problem",
      "correction": "I don't want any trouble",
      "context": "After the mugger's initial threat",
      "count": 2
    },
    {
      "user_said": "I am agree",
      "correction": "I agree",
      "context": "Responding to the mugger's demand",
      "count": 3
    }
  ],
  "hesitation_contexts": [
    {
      "context": "After the mugger raised his voice and demanded a faster answer"
    },
    {
      "context": "When asked to empty pockets — unfamiliar vocabulary"
    }
  ],
  "idioms": [
    {
      "expression": "Pull the other one",
      "meaning": "I don't believe you",
      "context": "The mugger used this when you claimed to have no wallet"
    }
  ],
  "areas_to_work_on": [
    "Negative sentence structure (use don't/doesn't instead of 'not want')",
    "Responding under pressure without freezing",
    "Using complete sentences instead of single-word answers"
  ],
  "inappropriate_behavior": null
}
```

### Schema Constraints

| Field | Type | Constraint |
|-------|------|-----------|
| `errors` | array | 0-5 items |
| `errors[].user_said` | string | User's exact words, verbatim from transcript |
| `errors[].correction` | string | The correct English form |
| `errors[].context` | string | 1 sentence max — when in the conversation |
| `errors[].count` | integer | Number of times this error occurred (>= 1) |
| `hesitation_contexts` | array | 0-3 items. Backend provides durations, LLM adds context |
| `hesitation_contexts[].context` | string | 1 sentence — what was happening when the user hesitated |
| `idioms` | array | 0-3 items. Hidden in UI if empty |
| `idioms[].expression` | string | The exact idiom/slang expression |
| `idioms[].meaning` | string | Plain English meaning |
| `idioms[].context` | string | When the character used it |
| `areas_to_work_on` | array | 2-3 strings. Theme + actionable parenthetical. Backend clamps: if LLM returns 1 item, display as-is; if 4+, truncate to first 3 |
| `inappropriate_behavior` | string or null | null if normal call. Factual explanation if FR37 triggered |

## Backend-Provided Fields (NOT generated by LLM)

| Field | Source | Notes |
|-------|--------|-------|
| `survival_pct` | Backend calculation | Integer 0-100. Based on scenario progression |
| `character_name` | Scenario config (DB) | e.g., "The Mugger" |
| `scenario_title` | Scenario config (DB) | e.g., "Give me your wallet" |
| `attempt_number` | `user_progress` table | Integer >= 1 |
| `previous_best` | `user_progress` table | Integer 0-100 or null (first attempt) |
| `hesitations[].duration_sec` | Backend (STT timestamps) | Float, measured by Soniox. Gaps > 3s only |
| `encouraging_framing.proximity` | Backend calculation | e.g., "5% away from surviving the mugger" |
| `encouraging_framing.improvement` | Backend calculation | e.g., "+6% since last attempt" |

**Assembly:** The backend merges LLM output + its own calculations to produce the complete debrief JSON sent to the client. The client receives the debrief ready to display.

**Hesitation merge logic:** The backend passes the top 3 gaps (>3s) to the LLM sorted by duration (longest first). The LLM returns `hesitation_contexts` in the same order. The backend merges by index: `hesitation_contexts[i].context` + backend-measured `duration_sec` → `hesitations[i]` in the client response. The field is renamed from `hesitation_contexts` (LLM) to `hesitations` (client) during assembly.

**Encouraging framing contract:** When `survival_pct ≤ 40`, the backend omits the `encouraging_framing` field entirely (per architecture convention: null fields omitted from JSON). The client checks for field presence to determine visibility. When `inappropriate_behavior` is non-null and survival > 40%, both sections coexist — they serve independent purposes (proximity data vs. call termination explanation).

## Section Ordering on Screen (Validated)

| # | Section | Visibility | Emotional Role |
|---|---------|-----------|----------------|
| 1 | **Hero** (%, character, scenario, attempt) | Always | **Impact** — the number hits. Screenshot-worthy. |
| 2 | **Encouraging Framing** (FR15b) | If survival > 40% | **Context** — data-driven proximity, not praise |
| 3 | **Language Errors** (FR10) | Always | **Analysis** — educational core, specific errors |
| 4 | **Hesitation Analysis** (FR12) | Always | **Behavioral insight** — when the user freezes |
| 5 | **Idioms & Slang** (FR13) | If idioms present | **Enrichment** — vocabulary encountered |
| 6 | **About This Call** (FR37) | If inappropriate behavior | **Explanation** — why the character hung up |
| 7 | **Areas to Work On** | Always | **Direction** — the user closes the app with this list |

**Reading arc:** Impact → Analysis → Direction.

## Design Impact Summary

The visual design spec (`debrief-screen-design.md`) requires these updates:

| Change | Detail |
|--------|--------|
| Error card: add `count` display | When `count >= 2`, display "x[N]" next to "You said:" label |
| Hesitation section: multi-card | Change from 1 card to 1-3 cards, add count indicator header |
| No `strengths` section | Confirms absence — no change needed |
| No `summary` section | Confirms absence — no change needed |
| Error card: keep `context` | No change — existing design is correct |

Layout, typography, spacing, colors — all remain identical. These decisions change the DATA MODEL, not the visual design.
