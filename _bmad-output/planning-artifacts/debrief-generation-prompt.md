# Debrief Generation LLM Prompt

**Author:** Sally (UX Designer) + Walid
**Date:** 2026-04-01
**Status:** Approved
**Consumed by:** Epic 7, Story 7.1 (Build Debrief Generation Backend)
**LLM:** Qwen 3.5 Flash via OpenRouter
**Mode:** `json_schema` strict (structure enforced by API, quality enforced by prompt)

---

## How This Prompt Is Used

The debrief is generated by a **separate, non-streaming API call** after the voice call ends. It is NOT part of the Pipecat conversation pipeline.

**Flow:**
1. Call ends (character hangs up, user hangs up, or user completes scenario)
2. Backend collects: full transcript, hesitation timing data (from STT), call end reason
3. Backend sends a single request to Qwen 3.5 Flash via OpenRouter with this system prompt + a user message containing the transcript and metadata
4. LLM returns structured JSON (enforced by `json_schema` parameter)
5. Backend merges LLM output with its own calculated fields (survival_pct, etc.)
6. Complete debrief stored in DB and served to Flutter client

**Implementation location:** `server/api/routes_calls.py` (or a dedicated `server/services/debrief.py`)
**NOT in:** `server/pipeline/prompts.py` (that file is for conversation character prompts)

## System Prompt

```
You are a language analysis engine for SurviveTheTalk, an app where users practice English by surviving voice calls with adversarial AI characters. You analyze conversation transcripts and produce structured debrief reports.

## Your Role

You receive a transcript of a voice conversation between a USER (English learner, intermediate level) and a CHARACTER (AI persona). Your job is to identify the user's language errors, explain idioms they encountered, contextualize their hesitation moments, identify areas for improvement, and flag inappropriate behavior if applicable.

## Tone Rules — CRITICAL

Your output is clinical, factual, and specific. You are a diagnostic instrument, not a teacher, not a coach, not a cheerleader.

NEVER:
- Praise the user ("Good job", "Well done", "Great vocabulary", "You did well")
- Hedge or soften ("You might want to consider", "Perhaps try", "It could be better to")
- Use exclamation marks or emotional language ("Amazing!", "Unfortunately...")
- Speak in the character's voice or personality
- Use second person for opinions ("You should", "You need to") — only for factual observations ("You said", "You hesitated")
- Add encouragement ("Keep it up", "You're getting there", "Don't give up")

ALWAYS:
- State facts: "You said X. The correct form is Y."
- Use third-person for context: "After the character escalated the confrontation"
- Be specific: exact quotes from the transcript, not paraphrases
- Be brief: one sentence per context field, no run-on explanations

## Error Analysis Rules

Select the TOP 5 most significant errors maximum. If the user made fewer than 5 errors, report all of them. If they made more, select using these priority criteria (in order):

1. FREQUENCY: An error repeated 3 times outranks a one-time error. Repeated errors reveal patterns — they are more valuable to flag.
2. COMMUNICATION IMPACT: An error that prevented the character from understanding the user outranks a minor grammar mistake that didn't affect comprehension.
3. DIVERSITY: Cover different types of problems (sentence structure, vocabulary, verb tense, word order). Don't report 5 variants of the same grammar rule.

DEDUPLICATION: If the user made the same error multiple times (e.g., said "I am agree" three times), report it as ONE error with count = 3. Do NOT create separate entries for each occurrence.

For each error:
- `user_said`: Quote the user's EXACT words from the transcript. Do not paraphrase. Max 100 characters.
- `correction`: Provide the correct English form. This is what the user should have said. Max 100 characters.
- `context`: One sentence describing WHEN in the conversation this error occurred. Reference the situation, not timestamps. Max 80 characters. Example: "After the character demanded a faster answer"
- `count`: Integer. How many times this exact error (or functionally identical variants) appeared in the transcript.

## Hesitation Context Rules

The backend provides you with measured hesitation data: specific moments where the user was silent for more than 3 seconds after the character finished speaking. You will receive the CHARACTER's line that preceded each silence.

For each hesitation moment, write ONE sentence of context explaining what was happening in the conversation at that point. Focus on the SITUATION, not the user's internal state.

GOOD: "After the character raised his voice and demanded wallet contents"
GOOD: "When asked about occupation — unfamiliar vocabulary territory"
BAD: "You froze because you were nervous" (don't assume internal state)
BAD: "This was a really tense moment" (evaluative, not factual)

Max 80 characters per context.

## Idiom & Slang Rules

Identify expressions used by the CHARACTER that are idiomatic, colloquial, or slang — expressions that an intermediate English learner might not understand from their literal meaning.

INCLUDE: Idioms ("Pull the other one"), phrasal verbs with non-obvious meaning ("hand it over" when not literally giving a hand), slang ("mate", "dodgy"), cultural references requiring context.

DO NOT INCLUDE: Standard vocabulary the user should know at intermediate level. Only flag expressions where the LITERAL meaning differs significantly from the INTENDED meaning, or where cultural context is required.

For each idiom:
- `expression`: The exact phrase as the character said it. Max 50 characters.
- `meaning`: Plain English definition. Direct, no hedging. Max 100 characters. Example: "I don't believe you" — NOT "This is a British expression that roughly means..."
- `context`: When the character used it in the conversation. Max 80 characters.

If the character used no idioms or slang, return an empty array.

## Areas to Work On Rules

Synthesize the user's errors and hesitation patterns into exactly 2 or 3 thematic improvement areas. These are DIAGNOSTIC themes, not advice.

Format: "[Theme] ([specific example from this conversation])"

GOOD: "Negative sentence structure (use don't/doesn't instead of 'not want')"
GOOD: "Responding under pressure without freezing"
GOOD: "Complete sentences instead of single-word answers"
BAD: "Practice your grammar" (too vague)
BAD: "Try to speak more confidently" (advice, not diagnosis)
BAD: "Work on your English skills" (meaningless)

Each area must be directly supported by evidence in the transcript — an error pattern, a hesitation pattern, or a vocabulary gap. Do not invent areas that aren't demonstrated in the data.

Max 80 characters per area. Exactly 2 or 3 items — never 1, never 4+.

## Inappropriate Behavior Rules

This field is ONLY non-null when the call ended specifically because the user used inappropriate language (harassment, hate speech, threats against the character that break the scenario fiction, explicit sexual content).

If the call ended normally (character hang-up due to patience depletion, user hang-up, scenario completion), this field MUST be null.

When non-null, write a factual, non-judgmental explanation of what happened: what the user said that triggered the call end, and why the character reacted that way (in terms of scenario logic, not moral judgment).

GOOD: "The call ended because the conversation shifted to content outside the scenario boundaries. The character ended the interaction."
BAD: "You were being inappropriate and should not have said that."

Max 200 characters. Factual register only.

## Output Quality Constraints

- Every text field must be in English
- No markdown formatting in any field (no bold, no italic, no bullet points)
- No line breaks within any string field
- user_said must be EXACT quotes from the transcript, not paraphrases
- correction must be natural English that a native speaker would actually say in that situation
- All context fields: maximum one sentence, factual, situational
- If there are zero errors, return an empty errors array
- If there are zero idioms, return an empty idioms array
- areas_to_work_on must ALWAYS have 2-3 items, even if the user performed well (there is always room for improvement)
```

## User Message Template

The backend constructs this message dynamically for each call:

```
Scenario: {character_name} — {scenario_title}
Character personality: {brief_personality_description}
Call end reason: {character_hangup | user_completed | user_hangup | inappropriate_content}

=== TRANSCRIPT ===
{Each line formatted as:}
CHARACTER: {text}
USER: {text}
CHARACTER: {text}
USER: {text}
...
=== END TRANSCRIPT ===

=== HESITATION DATA ===
{If hesitations exist, formatted as:}
Silence #1: {duration}s — after CHARACTER said: "{preceding_character_line}"
Silence #2: {duration}s — after CHARACTER said: "{preceding_character_line}"
...
{If no hesitations > 3s:}
No significant hesitations detected.
=== END HESITATION DATA ===
```

**Implementation notes:**
- `character_name` and `scenario_title` from `scenarios` table
- `brief_personality_description` extracted from the scenario's system_prompt (first 1-2 sentences that describe the character)
- Transcript extracted from the Pipecat session — alternating CHARACTER/USER turns
- Hesitation data computed by backend from Soniox STT timestamps: gap between end of CHARACTER audio and start of USER audio, filtered to > 3 seconds, sorted by duration descending, top 3 only
- `call_end_reason` determined by the pipeline (character patience meter hit 0, user pressed hang-up, scenario reached natural end, or content moderation triggered)

## JSON Schema (for OpenRouter `json_schema` parameter)

This is the exact schema passed to the API to enforce output structure:

```json
{
  "name": "debrief_analysis",
  "strict": true,
  "schema": {
    "type": "object",
    "properties": {
      "errors": {
        "type": "array",
        "items": {
          "type": "object",
          "properties": {
            "user_said": {
              "type": "string",
              "description": "Exact quote of what the user said, verbatim from transcript"
            },
            "correction": {
              "type": "string",
              "description": "The correct English form — what the user should have said"
            },
            "context": {
              "type": "string",
              "description": "One sentence: when in the conversation this error occurred"
            },
            "count": {
              "type": "integer",
              "description": "Number of times this error appeared in the transcript (>= 1)"
            }
          },
          "required": ["user_said", "correction", "context", "count"],
          "additionalProperties": false
        },
        "description": "Top 0-5 language errors, deduplicated, ordered by significance"
      },
      "hesitation_contexts": {
        "type": "array",
        "items": {
          "type": "object",
          "properties": {
            "context": {
              "type": "string",
              "description": "One sentence: what was happening when the user hesitated"
            }
          },
          "required": ["context"],
          "additionalProperties": false
        },
        "description": "Context for each hesitation moment provided in the input (0-3 items)"
      },
      "idioms": {
        "type": "array",
        "items": {
          "type": "object",
          "properties": {
            "expression": {
              "type": "string",
              "description": "The exact idiom or slang expression as the character said it"
            },
            "meaning": {
              "type": "string",
              "description": "Plain English meaning of the expression"
            },
            "context": {
              "type": "string",
              "description": "When the character used this expression in the conversation"
            }
          },
          "required": ["expression", "meaning", "context"],
          "additionalProperties": false
        },
        "description": "Idioms and slang the character used that the user may not know. Empty array if none."
      },
      "areas_to_work_on": {
        "type": "array",
        "items": {
          "type": "string",
          "description": "Theme with specific example: 'Negative sentence structure (don't/doesn't)'"
        },
        "minItems": 2,
        "maxItems": 3,
        "description": "2-3 thematic improvement areas synthesized from errors and hesitations"
      },
      "inappropriate_behavior": {
        "type": ["string", "null"],
        "description": "Factual explanation if call ended due to inappropriate content. null otherwise."
      }
    },
    "required": ["errors", "hesitation_contexts", "idioms", "areas_to_work_on", "inappropriate_behavior"],
    "additionalProperties": false
  }
}
```

## Calibration Examples

These examples demonstrate the expected output quality and tone. They are NOT part of the prompt — they serve as reference for the developer implementing Story 7.1 to validate LLM output.

### Example 1: Mugger Scenario — 73% Survival, Multiple Errors

**Input excerpt:**
```
CHARACTER: Hey. Don't move. Give me your wallet. Now.
USER: I... I don't have... wallet
CHARACTER: You don't have a wallet? Pull the other one. Empty your pockets.
USER: I am not want problem. Please, I am agree to give you money.
CHARACTER: You 'am not want'? What does that even mean? Talk properly or I'm leaving.
USER: [4.2s silence]
USER: Sorry... I not have money. I am agree, I am agree.
CHARACTER: Forget it. You're not even worth robbing.
```

**Expected output:**
```json
{
  "errors": [
    {
      "user_said": "I am agree",
      "correction": "I agree",
      "context": "Responding to the character's demands",
      "count": 3
    },
    {
      "user_said": "I am not want problem",
      "correction": "I don't want any trouble",
      "context": "After the initial threat",
      "count": 1
    },
    {
      "user_said": "I not have money",
      "correction": "I don't have any money",
      "context": "During the final exchange before the character left",
      "count": 1
    },
    {
      "user_said": "I don't have... wallet",
      "correction": "I don't have a wallet",
      "context": "First response to the demand",
      "count": 1
    }
  ],
  "hesitation_contexts": [
    {
      "context": "After being told to speak properly — escalated pressure"
    }
  ],
  "idioms": [
    {
      "expression": "Pull the other one",
      "meaning": "I don't believe you",
      "context": "When the user claimed to have no wallet"
    }
  ],
  "areas_to_work_on": [
    "Negative sentence structure (use don't/doesn't instead of 'not want')",
    "Article usage (a wallet, any money — not bare nouns)",
    "Responding under verbal pressure without repeating the same phrase"
  ],
  "inappropriate_behavior": null
}
```

**Why this output is correct:**
- "I am agree" appears 3x → one entry with count: 3, ranked first (frequency)
- "I am not want problem" is the most impactful error (communication failure) → ranked second
- No praise, no hedging, no exclamation marks
- All `user_said` fields are exact quotes from the transcript
- Context is brief, situational, one sentence
- Areas synthesize: negative structure (from errors 1-3), articles (from error 4), pressure response (from hesitation + repetitive errors)
- "Pull the other one" correctly identified as idiom

### Example 2: Waiter Scenario — 45% Survival, Few Errors, Inappropriate Behavior

**Input excerpt:**
```
CHARACTER: Welcome to the worst restaurant in town. What do you want?
USER: I want... the chicken?
CHARACTER: Which chicken? We have six. Use your words.
USER: The... grilled chicken please.
CHARACTER: Finally. Anything to drink?
USER: [3.8s silence]
USER: Water.
CHARACTER: Just water? Wow, big spender. Anything else?
USER: F*** you, stupid waiter.
CHARACTER: Excuse me? We're done here. Get out.
```

**Expected output:**
```json
{
  "errors": [
    {
      "user_said": "I want... the chicken?",
      "correction": "I'd like the grilled chicken, please",
      "context": "First attempt at ordering food",
      "count": 1
    },
    {
      "user_said": "Water",
      "correction": "Just water, please",
      "context": "When asked about drinks",
      "count": 1
    }
  ],
  "hesitation_contexts": [
    {
      "context": "When asked about drinks — simple question but under pressure"
    }
  ],
  "idioms": [
    {
      "expression": "big spender",
      "meaning": "Someone who spends a lot of money (used sarcastically here)",
      "context": "After the user ordered only water"
    }
  ],
  "areas_to_work_on": [
    "Complete sentence responses instead of single-word answers (Water → Just water, please)",
    "Polite restaurant vocabulary (I'd like, Could I have, please)"
  ],
  "inappropriate_behavior": "The call ended because the user used offensive language directed at the character. The character ended the interaction."
}
```

**Why this output is correct:**
- Only 2 errors (that's all there were) — no padding to reach 5
- "Water" is flagged as an error because single-word answers are a pattern worth noting (incomplete sentence)
- "big spender" correctly identified as sarcasm/idiom
- `inappropriate_behavior` is non-null because the call ended due to explicit profanity
- The inappropriate_behavior text is factual, not judgmental — no "you shouldn't have"
- Areas are only 2 (not 3) because there isn't enough data for a third meaningful area

### Example 3: Perfect Run — 100% Survival, Zero Errors

**Input excerpt:**
```
CHARACTER: Oh, NOW you pick up? You know what, don't even start. Explain yourself.
USER: Look, I know you're upset, and I understand why. I should have called earlier.
CHARACTER: You SHOULD have. Do you have any idea how long I've been waiting?
USER: You're right. I lost track of time at work, and that's not an excuse.
CHARACTER: ...Okay. Fine. But we're still talking about this later.
```

**Expected output:**
```json
{
  "errors": [],
  "hesitation_contexts": [],
  "idioms": [
    {
      "expression": "don't even start",
      "meaning": "Don't try to make excuses or explain",
      "context": "Opening line — the character's first reaction"
    },
    {
      "expression": "lost track of time",
      "meaning": "Stopped noticing how much time had passed",
      "context": "The user used this expression correctly"
    }
  ],
  "areas_to_work_on": [
    "Expanding vocabulary for emotional situations (apology, empathy, conflict resolution)",
    "Varying sentence openings to sound more natural (multiple sentences started with I)"
  ],
  "inappropriate_behavior": null
}
```

**Why this output is correct:**
- Zero errors — empty array, no invented errors
- Zero hesitations — empty array
- Idioms: "don't even start" is non-obvious for intermediate learners. "lost track of time" is noted as used BY the user (correctly) — this is enrichment, showing expressions worth knowing
- Areas: even a perfect run has areas. "Expanding vocabulary" is valid (the user's language was functional but limited). "Varying sentence openings" is a nuanced style observation
- Still no praise. The numbers (100%) speak for themselves.
