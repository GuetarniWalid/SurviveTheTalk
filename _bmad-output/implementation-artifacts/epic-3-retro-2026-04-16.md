## Executive Summary

Epic 3 delivered 3/3 stories (Stories 3.0, 3.1, 3.2) but required a **major mid-epic scope pivot** (2026-04-15 sprint change proposal): the scenario system migrated from monolithic system prompts to a checkpoint-based progression system. Team handled the pivot cleanly — zero code thrown away, 11 artifact conflicts documented and approved, 2 new downstream stories (6.6 CheckpointManager, 6.7 CheckpointStepper) created. The Waiter scenario was validated end-to-end on the live VPS pipeline; the 4 remaining scenarios (Mugger, Girlfriend, Cop, Landlord) shipped as YAML scaffolds with calibration **formally deferred to post-Epic 6** (when CheckpointManager exists to compute meaningful survival %). 5 technical debt items (D1–D5) were documented proactively for Epic 6 consumption. The retrospective surfaces three systemic patterns: (1) planning gap allowed the monolithic assumption to propagate 5 scenarios deep before correction, (2) sprint-status discipline remains imperfect (epic-3 still `in-progress` at retro start), (3) Epic 4 — the first code-heavy MVP epic — starts with meaningful unfinished housekeeping from Epic 1 & 2.

## Successes

### 1. Mid-Epic Scope Pivot Handled Cleanly
The 2026-04-15 sprint change proposal documented 11 artifact conflicts (3 architecture, 5 UX, 5 difficulty-calibration) with incremental approval. Impact analysis correctly identified zero code wasted (Epic 3 was content-only, downstream epics not started). Two new stories (6.6 CheckpointManager, 6.7 CheckpointStepper) added to Epic 6 with full specs. This is what disciplined course correction looks like.

### 2. Checkpoint Architecture Is a Real Quality Win
The pivot was not noise — it fundamentally improved the product. Content-based scoring (`checkpoints_passed / total_checkpoints`) replaces the speech-count heuristic that could not distinguish a performing user from one saying irrelevant things. The CheckpointStepper overlay adds real-time user guidance without violating the "no system text" UX principle (treated as gameplay HUD, not chrome).

### 3. Calibration Tools Built Once, Useful Twice
Story 3.0 produced `TranscriptLogger` and `score_transcript.py` with 44 new tests (54 total). Even though the full calibration pass is deferred, the tools already paid off: they uncovered the TranscriptLogger `super().process_frame()` bug and the VAD interruption cascade during Waiter validation. Tools that surface production bugs before MVP are worth their weight.

### 4. Waiter Scenario Validated End-to-End on Live VPS
Not a lab test — the full Pipecat pipeline (Soniox STT → Qwen LLM → Cartesia TTS) was exercised on the VPS with two real voice calls. Character feel was iterated (tired-not-angry rewrite), VAD parameters tuned (stop_secs 0.8, speech_timeout 1.8). This proves the pipeline is production-capable.

### 5. Tech Debt Documented Proactively, Not Discovered Later
5 Epic 6 tech debt items (D1–D5) were captured during Story 3.0 code review. All flag real issues: `/tmp` permissions, token-level TextFrame fragmentation, race condition on shared state, crash recovery, session ID collision. Epic 6 can now plan with eyes open instead of rediscovering these at integration time.

### 6. Adversarial Code Review Is Working
Story 3.1 received 12 corrections from adversarial review (Blind Hunter + Edge Case Hunter + Acceptance Auditor). Story 3.2 received pipeline-validation fixes. Multiple review layers caught issues that would have shipped otherwise. The process defined in Epic 2 is now operational.

### 7. Ordered Intent Preserved Across Characters
All 5 scenarios share identical YAML structure (metadata / base_prompt / checkpoints / exit_lines / briefing / calibration) while each character has distinct personality content. Consistency discipline held even under scope-change pressure.

## Challenges

### 1. Monolithic Prompt Assumption Propagated Too Far Before Correction (Critical Planning Gap)
The first draft of 5 scenarios was written in monolithic format before the checkpoint-based approach emerged. The `survival_pct = successful_exchanges / expected_exchanges` formula did not differentiate a performing user from one saying irrelevant things — this should have been caught during Epic 3 planning or during the PRD/architecture review, not during Story 3.2 execution.

**Root cause:** The scoring model was specified as a formula without ever being traced through a concrete example of "user says 'banana banana banana' six times." Would have been caught by one adversarial walk-through.

**Impact:** Story 3.1 reopened, Story 3.2 Phase 1 redone. Net effort recovered via checkpoint quality gains, but the rework was preventable.

**Resolution:** Sprint change proposal executed cleanly. Action item: for code-heavy epics (Epic 4+), add a "concrete user scenario walk-through" step to story creation.

### 2. 4 of 5 Scenarios Calibration-Deferred to Post-Epic 6
Only The Waiter was validated end-to-end. The remaining 4 (Mugger, Girlfriend, Cop, Landlord) have complete YAMLs but their `calibration.pass_a` and `calibration.pass_b` blocks are placeholders. Their `tts_voice_id` is `null` (voice selection deferred). Meaningful survival % measurement requires the CheckpointManager (Epic 6 Story 6.6) to track checkpoint progression — without it, the LLM improvises and survival % is just utterance count.

**Root cause:** Technically unavoidable — checkpoint-based scoring cannot be tested without the checkpoint machinery. The decision to defer was correct.

**Impact:** Epic 3's original AC "60-80% survival on first scenario" cannot be formally validated until post-Epic 6. This is a **latent acceptance criterion** that must not be forgotten.

**Resolution:** Action item (P0): track "Post-Epic 6 Calibration" as a formal follow-up deliverable, visible in sprint-status or a dedicated tracking document. Must not be "remembered" via retro doc only.

### 3. Sprint-Status Discipline Still Imperfect (Third Epic in a Row)
At the start of this retro, `epic-3` was still marked `in-progress` despite all 3 stories being `done`. This is the third consecutive epic where the epic-level status was not updated.

**Root cause:** No automation, no checklist trigger. Dev and reviewer both rely on memory.

**Impact:** Sprint tracking inaccurate. Walid manually corrects after the fact.

**Resolution:** Retrospective workflow now updates `epic-N: done` as part of the retro save (action item). Longer-term: add to the "Last bullet before commit" checklist.

### 4. Pipeline Bugs Discovered in Production, Not in Dev
Three bugs surfaced only during live VPS testing:
- TranscriptLogger missing `super().process_frame()` silently blocked the entire Pipecat pipeline
- VAD interruption cascade required parameter tuning (discovered on first call)
- systemd private tmp path (`/tmp/systemd-private-*/tmp/...`) not the vanilla `/tmp` — broke transcript retrieval

**Root cause:** No pre-production smoke test harness. Unit tests validated the class in isolation but not the end-to-end frame flow.

**Impact:** Each bug required SSH → fix → restart → retry loop during calibration. Time cost was real.

**Resolution:** For Epic 4+, define a "compile + deploy + round-trip call" smoke gate per story touching the live pipeline. Cheap insurance.

### 5. Intention UX Section Applied to 1 of 3 Stories
Epic 2 action item #2: "Add Intention UX section to every story." Story 3.0 has it. Story 3.1 and 3.2 do NOT. 33% adherence.

**Root cause:** No template enforcement during `create-story`. Relies on memory.

**Impact:** Intention UX is high-value for code-heavy stories (Epic 4+) where implementation choices affect user feel. Missing the discipline now means missing it when it matters.

**Resolution:** Action item: bake Intention UX section into the create-story template before starting Epic 4.

### 6. Known Issues Mapping (PoC → MVP Design) Still Not Done
Carried forward from Epic 1 → Epic 2 → Epic 3. Was "not blocking" for Epic 2 and Epic 3 (design/content epics). Becomes **blocking** for Epic 4 — the first epic where PoC known issues (cold start, debrief latency, silence handling) will directly hit MVP code.

**Root cause:** Always easier to defer. No forcing function.

**Impact:** If still not done at Story 4.1 start, we will discover PoC issues mid-MVP-build, costing rework.

**Resolution:** Action item (CRITICAL PATH): complete the mapping BEFORE Story 4.1 begins.

## Key Insights

1. **Scope pivots during content epics have near-zero cost. Do not hesitate when the product direction is wrong.** The Epic 3 pivot cost ~3 planning artifacts and 5 scenario rewrites. If it had been caught in Epic 5 or 6 with code already written, cost would have been 10×+.

2. **Specifications must be traced through a concrete adversarial example.** A formula or rule that looks right on paper ("user speaks N times = success") can fail a single walk-through ("what if user says 'banana'?"). Add this step to story creation for Epic 4+.

3. **Deferred calibration is still owed work. Track it as a first-class deliverable.** "Post-Epic 6 Calibration" is not a retro footnote — it is a real acceptance criterion for the product to ship. Make it visible.

4. **Live pipeline bugs are always deeper than unit tests reach.** Unit tests caught zero of the three Waiter bugs. A single "deploy + one round-trip call" smoke test would have caught all three.

5. **Carried-forward action items accumulate interest.** The PoC known issues mapping has carried for two retros. Third time blocking. Rule: an action item with a next-epic deadline must have a forcing function (critical path item in sprint-status), not just a retro-doc line.

6. **Sprint-status discipline is not a memory problem. It is a process problem.** Three consecutive epics forgetting the epic-level update means the process is wrong, not the people. The retrospective workflow now owns the epic-N close-out.
