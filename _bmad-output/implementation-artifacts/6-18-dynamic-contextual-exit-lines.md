# Story 6.18: Dynamic / contextual character hang-up + warning lines

Status: ready-for-dev

> Design decisions RESOLVED with Walid 2026-06-03 ‚Äî see `## Decisions`. Final version, ready for `/bmad-dev-story`.

## Story

As the learner,
I want the character's final words (when it hangs up on me, or warns me it's losing patience) to **reflect what I actually said and how the call actually went**,
so that the ending feels coherent and real ‚Äî not a canned accusation that contradicts my conversation.

## Background

**The trigger ‚Äî cop call `call_id=212` (2026-06-03).** The cop hung up after the user met only 2/20 checkpoints (patience drained ‚Äî the scenario is `hard` and linguistically demanding) and spoke the **hardcoded** `exit_lines.hangup`: *"That's the second time your story's changed on me. Innocent people don't need three versions‚Ä¶"* ‚Äî even though the user **never gave a single version/alibi**. The accusation is a canned YAML string that fires on **any** `character_hung_up`, regardless of the real failure mode ‚Üí incoherent.

**Root cause.** Exit lines (and the patience-warning line) are static text loaded from the scenario YAML (`exit_lines.hangup` / `exit_lines.completion` / `exit_lines.noisy_environment` / `patience_warning`) and selected purely by the hang-up *reason*. They are written assuming an "ideal" failure (the user told contradictory stories), but the engine plays the same line whether the user contradicted themselves, gave nothing, went silent, or actually succeeded. The **`COHERENCE_CHARTER`** (Story 6.8) governs every *generated* character turn so it can't invent facts ‚Äî but it does **not** cover these hardcoded lines.

**The fix (Walid 2026-06-03):** make the closing (and warning) line **dynamic** ‚Äî generated in-character by the character LLM from the **actual conversation + the hang-up reason**, so it can only reference what really happened. Keep the YAML `exit_lines` as a **fast fallback** for latency/failure. This closes the coherence gap on the one part of the call the charter doesn't reach. See `memory/project_dynamic_hangup_and_difficulty_selector.md` and `[[feedback_coherence_must_be_system_wide]]`.

**Out of scope:** changing the difficulty (the cop stays `hard` ‚Äî Walid wants the challenge; the difficulty *selector* is the separate Story 6.19). This story is coherence only ‚Äî the character will still hang up just as fast, but with words that match the call.

## Acceptance Criteria (BDD)

### AC1 ‚Äî The hang-up line is generated from the real conversation
Given a call ends via `_run_hang_up(reason)` (reason ‚àà `character_hung_up`/silence, `inappropriate_content`, `noisy_environment`, `survived`)
When the exit line is produced
Then it is **generated by the character LLM** from (the conversation transcript via `llm_context.get_messages()` + the hang-up reason + the persona/`base_prompt` + the `COHERENCE_CHARTER`), so it references only what the user actually said and **never fabricates events that did not occur** (e.g. it must NOT say "you changed your story" unless the user actually gave conflicting versions).

### AC2 ‚Äî Canned line is the guaranteed fallback
Given generation can fail, time out, or be disabled
When the generated line is not available within the latency budget
Then the existing YAML `exit_lines.<reason>` (or its hardcoded default) is spoken instead ‚Äî **there is always a final line**, never silence or a crash. Generation failure is logged at WARNING and swallowed (mirror `llm_warmup.py` fire-tolerance).

### AC3 ‚Äî Latency-bounded, ‚â§2 sentences
Given the hang-up sequence already waits up to `_HANG_UP_TTS_TIMEOUT_SECONDS = 6.0s` for the exit-line TTS (`patience_tracker.py:158`)
Then the generation step is **time-boxed** (default ‚â§1.5s) so the full hang-up sequence stays within budget, and the generation prompt constrains the line to **‚â§2 sentences** (a longer line risks the 6s TTS ceiling). If the box elapses, AC2 fallback fires.

### AC4 ‚Äî Covers every reason + the patience warning
Given the four hang-up reasons + the separate one-shot `patience_warning` (`patience_tracker.py:_emit_patience_warning`, fired when the meter enters the warning band)
Then each path produces a **reason-appropriate** generated line: `character_hung_up`/silence ‚Üí "you've gone quiet / I'm not getting straight answers"; genuine contradictions in the transcript ‚Üí may reference them; `inappropriate_content` ‚Üí in-persona shutdown; `survived` ‚Üí in-character positive closure; `patience_warning` ‚Üí an in-context "last chance" tied to what's actually missing. The reason + transcript drive the wording.

### AC5 ‚Äî Coherence-by-construction (the 212 regression, as a permanent guard)
Given the cop-212 failure (accused "3 versions" with zero versions given)
Then a test drives a hang-up where the transcript contains **no contradictions** and asserts the generated line does **not** claim the user changed/contradicted their story; and the universal rule (no inventing facts) is enforced via the `COHERENCE_CHARTER` in the generation prompt.

### AC6 ‚Äî `noisy_environment` interruption preserved
Given `noisy_environment` is the only reason that can fire while the character is mid-generating a normal reply (Story 6.11 call_id=200), pushing an `InterruptionFrame` before the exit line (`patience_tracker.py:1039-1043`)
Then the generated-line path preserves that ordering (interrupt ‚Üí then the generated/fallback line), and does not re-introduce the queued-behind-a-reply bug.

### AC7 ‚Äî Observability + safety toggle
Given operators must confirm the feature works + be able to kill it fast
Then each hang-up logs whether the line was **generated** vs **fallback** + the generation latency (journalctl-greppable), and an env toggle (e.g. `HANGUP_LINE_GENERATION=1|0`, default decided in `## Decisions`) flips the whole feature back to canned lines with no redeploy of logic.

### AC8 ‚Äî Cost / determinism (don't break `pytest`)
Given the generation is a live LLM call
Then the generation logic has fast **mocked-LLM unit tests** that run in the default `pytest` (generation success, timeout‚Üífallback, failure‚Üífallback, ‚â§2-sentence prompt, no-contradiction guard), and **no live LLM call runs in `pytest`**.

### AC9 ‚Äî Pre-commit gates
`ruff check . && ruff format --check . && pytest` (server) green; existing hang-up/patience tests still pass (the 4-reason selection + survival_pct deviation must be unchanged); `flutter` untouched (server-only story).

### AC10 ‚Äî Validation (smoke gate)
See `## Smoke Test Gate` ‚Äî a device call that ends in a hang-up must produce a final line that matches the conversation (the cop no longer says "3 versions" when none were given).

- `bot.py:382-405` ‚Äî PatienceTracker construction (where to inject the generator callable). `bot.py` has the `llm` (OpenAILLMService) + `llm_context`.
- `checkpoint_manager.py:_last_character_line` (~873-897) ‚Äî the existing pattern for reading `llm_context.get_messages()` (the transcript source).
- `prompts.py` ‚Äî `COHERENCE_CHARTER` (66-108), the place for the new generation prompts. `llm_provider.py` ‚Äî `resolve_llm_chat_url`/`resolve_llm_api_key`, `CHARACTER_TEMPERATURE` (0.7) / `CHARACTER_MAX_TOKENS` (256, use a smaller cap for one line).
- `llm_warmup.py` ‚Äî the **canonical standalone-LLM-call pattern** to mirror (httpx, fire-and-forget, never raises).

**Gotchas (do NOT trip these):**
- Generated line **‚â§2 sentences** or the 6s TTS timeout can fire ‚Üí WARNING + force-proceed.
- `survival_pct` differs by reason (Deviation #1): `survived`‚Üí100, others‚Üímeter-ratio (`:1079-1096`). **Do not touch this** ‚Äî the generated line is text only; the envelope math is unchanged.
- `noisy_environment` is the only path that interrupts an in-flight reply; keep the InterruptionFrame **before** the (generated) line.
- The hang-up must NEVER crash on a generation error ‚Äî a late/canned line beats no line. The whole sequence is fire-and-forget from `_schedule_hang_up`; tolerate a closed httpx client / missing context during teardown (check the `_hang_up_in_progress` guard).
- Observe **finalized** transcriptions only when reading context (`getattr(frame,'finalized',False)` ‚Äî server/CLAUDE.md ¬ß1).
- The character LLM is `OpenAILLMService` pointed at Groq base_url ‚Äî never import `pipecat.services.groq` (needs the uninstalled `groq` SDK). Use the `llm_provider` resolvers.

**Reuse:** `llm_warmup.warm_up_llm` (standalone call), `exchange_classifier._parse_*` (response parsing/validation), the `_run_hang_up` reason-switch, the PatienceTracker "accept-store-validate kwarg" pattern, `test_bot_pipeline_wiring`/`test_patience_tracker` constructor-injectable mocks.

## Decisions (RESOLVED 2026-06-03)

1. **Generation timing** ‚Üí **BLOCKING ~1.5s + canned fallback** (Walid: oui). The line is generated before the TTS push; if the ~1.5s time-box elapses (or generation fails), the canned YAML line is spoken. 1.5s sits comfortably inside the 6s hang-up TTS window.
2. **Patience-warning line** ‚Üí **also dynamic** (Walid: oui) ‚Äî same generator, "warning" variant (fixes the call-212 case where the warning asked "where were you at 8:30?" before the conversation had reached the alibi).
3. **Toggle default** ‚Üí **ON** (Walid: oui); `HANGUP_LINE_GENERATION=0` flips the whole feature back to the canned lines with no logic redeploy.
4. **`survived` (success) line** ‚Üí **also dynamic** (Walid: dynamique) ‚Äî generate the closing line for **all** reasons, including `survived`. The small extra latency on a winning ending is accepted for the coherence/quality win (the canned fallback still applies if generation is slow).
