## Successes & Strengths

### 1. Active enforcement layer for migration safety (win of the epic)

Story 5.1's first VPS deploy crashed on migration 003 (`FOREIGN KEY constraint failed`) because the local test DB was empty: no rows in `users` to violate FK during the table-rebuild. Local pytest was 100% green, prod was broken on landing.

Walid pushed back on the obvious "add a checklist / write a doc" reflex and asked for an *active* enforcement layer. Plan agent rejected a draft of 5 passive doc layers in favor of 1 active test layer:

- `server/scripts/refresh_prod_snapshot.py` — SSHs to VPS, sanitises PII (emails → `user-{id}@example.invalid`, jwt_hash → NULL, auth_codes → deleted), commits ~100 KB snapshot
- `tests/fixtures/prod_snapshot.sqlite` — sanitised live DB, tracked
- `tests/test_migrations.py` — replays migrations + full FastAPI lifespan against the snapshot, asserts `PRAGMA foreign_key_check` + `integrity_check` clean + no row-count drift

Demonstrated correctly: a buggy 003 (without `PRAGMA foreign_keys=OFF/ON`) raises `IntegrityError` against the snapshot — would have failed pytest locally before any deploy. Same approach replayed cleanly through 005-006.

**Why this matters:** the regression class "test passes on empty DB / fails on prod shape" is now structurally impossible to ship. CLAUDE.md was updated with a "Database Migrations — Test Against Production Shape" section as the active rule.

## Challenges & Growth Areas

### 1. Production regression on Story 5.1 (the lesson that became the win)

First VPS deploy of migration 003 crashed: `FOREIGN KEY constraint failed`. Local tests were 100% green because the test DB was empty — no `call_sessions` rows referenced the `users` rows that the table-rebuild was DROP-then-recreate-ing.

The fix itself was a one-liner (`PRAGMA foreign_keys = OFF/ON` around the table rebuild — standard SQLite idiom). The cost was a deploy + rollback cycle on production. The *win* came from Walid not stopping at the fix and instead asking for the active guardrail. **The regression was the catalyst, not the failure.**

This is the kind of thing project memory should hold permanently — `feedback_sqlite_table_rebuild_fk.md` already records it.

### 2. Auth 401 silent loop — flagged HIGH-PRIORITY, MUST-FIX before MVP launch

Surfaced in Story 5.5 review (2026-04-29). When the JWT expires, the server returns HTTP 401 with `code: 'AUTH_UNAUTHORIZED'`. The client has **no Dio interceptor** — every feature bloc that hits the API translates this through `_classifyApiCode` and the user sees a generic "try again". Retrying does nothing (token is dead). User is stuck in a loop until they kill and restart the app, and even then there's no graceful session-expiry re-auth flow.

Walid flagged this explicitly during review with "ne surtout pas oublier" — a non-trivial fraction of returning users will hit a dead-end loop on day-30+ and churn silently.

**Why deferred (not bad behavior):** the right fix is architecturally cross-cutting — a Dio interceptor in `client/lib/core/api/api_client.dart` that catches 401, clears `TokenStorage`, dispatches `AuthEvent.SessionExpired`, and lets `GoRouter` redirect to `/auth` via `refreshListenable`. Coupling 401 handling into `ScenariosBloc` (or any single feature bloc) would leak the auth abstraction and multiply maintenance. It deserves its own story.

**Where it lives:** top of `deferred-work.md` under "MUST-FIX BEFORE MVP LAUNCH" + project memory `feedback_auth_401_gap.md`. Action item AI-1 below.

### 3. Back-press blocking on `/call` — four attempted fixes, all failed

Surfaced in Story 5.2 review (2026-04-27). Swiping back from the call screen on Android does NOT bounce off `PopScope(canPop: false)` — system goes straight to `moveTaskToBack` and the app drops to background. Verified on a Pixel 9 Pro XL via on-device adb logcat tracing.

Tried during the review session:

1. `PopScope(canPop: false)` on the Scaffold root — kept (harmless), but on its own the Flutter back-callback flapped between Flutter's and Android's default callbacks during predictive back gestures.
2. Manifest `android:enableOnBackInvokedCallback="true"` — opted into Android 13+ predictive back. Did not change flapping behaviour.
3. Native `OnBackInvokedDispatcher.registerOnBackInvokedCallback(PRIORITY_OVERLAY, ...)` from `MainActivity.kt` — registered overlay callback was apparently never invoked. Either silent registration failure or Flutter's `FlutterActivity` overrides the dispatcher.
4. AndroidX `OnBackPressedDispatcher.addCallback(...)` — required switching `MainActivity` from `FlutterActivity` to `FlutterFragmentActivity`. Same backgrounding behaviour after rebuild.

**Probable root cause:** known interaction between Flutter's embedding, `go_router`'s `CustomTransitionPage`, the `_GoRouterRefreshStream(authBloc.stream)` listenable, and Android's predictive back system. The `PopScope` (or any back interceptor) is being torn down or de-prioritised at the moment the gesture commits.

**Decision (Walid 2026-04-27):** revert all native + manifest changes; keep only `PopScope(canPop: false)` (harmless); defer real fix to Story 6.1 with CallKit (iOS) + ConnectionService (Android). Those native call APIs own the audio session lifecycle AND back-press behaviour at the OS level — back-swipe is naturally blocked while the call is "ongoing" because the OS treats it as a system-managed phone call.

**Why this is acceptable for shipping Epic 5:** the `/call` route is still a placeholder; real WebRTC + Rive call screen is Story 6.1+. Backgrounding cleanly calls `room.disconnect()` — no resource leak. Cold-start lands on `/scenarios` — no broken state. Sub-optimal UX but not a regression.

Action item AI-2 below.

### 4. Rive single-instance constraint discovered the hard way

Story 5.2 first iteration replaced flat avatar circles with `CharacterAvatar` Rive puppets in the scenario list. On-device run crashed with `SIGSEGV` in `libGLES_mali.so` after rendering the list. Root cause: `RLIMIT_NOFILE` exhaustion. Each `RiveWidget` allocates its own Vulkan/EGL swapchain (~30-50 fd via sync fences + `ImageReader` buffers); five 50×50 puppets in parallel pushed the process past Android's ~1024 fd ceiling, causing the Mali driver to dereference null in `eglp_color_buffer_get_surface_template`.

Logs: `EGL_BAD_ALLOC` → `sync_merge ... Too many open files (-24)` → `Fatal signal 11 (SIGSEGV) ... in tid mali-event-hand`.

**Fix:** swapped `CharacterAvatar` (Rive) inside `ScenarioCard` for `Image.asset` + `ClipOval` using static 512×512 JPGs. The animated Rive puppet stays on the in-call screen (single instance — never crashed). Five JPG assets shipped (`waiter`, `mugger`, `girlfriend`, `cop`, `landlord`).

**Why this matters for Epic 6:** Story 6.2 (Call Screen with Rive Character Canvas) IS a single Rive instance — safe. But the constraint must be remembered: never ship N parallel `RiveWidget`s on the same screen. Action item AI-5 below.

## Key Insights & Learnings

1. **Active enforcement layer beats passive documentation.** When a regression class can be caught structurally (snapshot-replay test), do that — even if it costs ~100 KB of fixtures and an SSH script. CLAUDE.md sections that codify *running* checks (not norms to remember) are doing the most work.
2. **Adversarial review is paying for itself, story over story.** Healthy dismissed-as-noise rate (33 of 58 raw findings on 5.5) means reviewers are aggressive enough. Same-day patch application keeps the loop tight.
3. **Deferred-work items close opportunistically when triggers fire.** 5.3's TOCTOU patch organically resolved 5.1's deferred busy_timeout — because the new contending writer made the latent bug observable. The "Trigger check" line in deferred-work.md is doing real work; keep writing them.
4. **Story spec amendments belong inside the story file, not a separate change log.** The strikethrough + dated AMENDED markers in 5.4/5.5 are the right shape: future readers see both the intent and the shipped reality. Don't move this to git history alone.
5. **Cross-cutting concerns must NOT be folded into single-feature blocs.** Auth 401 in `ScenariosBloc`'s error mapping was rejected for the right reason — leaks abstraction, multiplies maintenance. Hold the line on this in Epic 6 (silence/hang-up/network-loss/call_end will tempt similar shortcuts).
6. **One Rive widget per visible surface.** RLIMIT_NOFILE is real on Android. Document the constraint, point Story 6.2 at it, never ship N parallel `RiveWidget`s again.
7. **Velocity step-up is structural, not heroic.** 6 days for 5 stories is the compounding return on Epic 1-4's investments (story spec rigor, design tokens, gotchas doc, smoke-test gate, migration guardrail). This is the cost curve we want to keep.
